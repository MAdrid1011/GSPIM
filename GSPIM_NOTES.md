# GSPIM：一种具有近存时间贡献筛选和跨帧高斯渲染流的4DGS 实时渲染异构架构

## Introduction

## Background

### 4D高斯泼溅

动态场景新视图合成的核心目标在于基于多视角视频数据，重建随时间变化的三维空间结构，并能够在任意时间点与视角下实时生成具备照片级质量的图像，实现完整且连续的视频渲染。该任务在虚拟现实、远程会议、机器人感知及自动驾驶等领域具有重要的理论价值与实际应用意义，同时对渲染质量与延迟性能提出了极高的要求。自3D高斯泼溅（3DGS）方法问世以来，显式高斯云作为一种兼具高渲染质量与实时性的三维静态场景表示方式，受到了广泛关注。具体而言，静态场景被建模为一组三维高斯基元，每个基元包含空间均值、协方差矩阵以及与视角相关的球谐函数（SH）颜色和不透明度信息，借助GPU友好的光栅化管线实现高效的新视图合成。基于3DGS的优势，4D高斯泼溅（4DGS）方法将该思想从三维静态空间扩展至包含时间维度的四维时空体积，将动态场景视为随时间演变的四维高斯分布集合。相比于3DGS对静态场景的重建，4DGS更侧重于在时间维度上对连续帧进行连续推理与渲染，从而实现动态视频的高质量重建。

近年来，4DGS技术迅速发展，涌现出多种可归类为4DGS的方法。总体而言，这些方法可分为三大类：（i）基于3DGS并结合时间偏移或速度场扩展的方法（如Dynamic 3D Gaussians、Gaussian Flow、Velocity-GS、Appearance Flow GS）。此类方法本质上仍以静态三维高斯为基础，通过附加运动场实现动态变化，通常依赖深度神经网络（尤其在复杂运动场景下）预测随时间变化的偏移量、尺度变化或外观变化，且对内存访问需求较高；（ii）基于时空分解的四维特征结构方法（如4D-KPlanes、HexPlane四维版本、Plenoxels-4D、Tensor4DGS）。此类方法通过因式分解或多平面结构，借助复杂的重建网络在四维空间中构建低秩场，将四维体积数据逐帧分解为多个三维或二维特征块，进而解码为三维高斯参数或颜色密度；（iii）纯粹的四维高斯场建模方法，直接在四维连续空间（x, y, z, t）上定义单个四维高斯基元，利用四维均值向量、$4\times 4$协方差矩阵及四维旋转参数完整描述其时空支持特性。

相较于方法（i）和（ii），从体系结构角度来看，纯4D高斯场建模在视频重建中展现出两大显著优势：首先，通过统一的时空集合进行建模，避免了复杂的时间场预测过程，无需进行点级别的预测与追踪，更好地契合现有计算平台（如GPU）对批量处理的硬件需求；其次，其算子结构高度简洁，所有时间相关算子均可归纳为小维度矩阵运算、一维高斯分布的解析计算及简单的乘加操作，极大地适合硬件加速实现。因此，该建模方法更符合当前硬件处理特性，在众多4D高斯场算法中具有代表性与优势。

### 4D 高斯泼溅数据流

相较于仅对三维空间分布进行建模的静态三维高斯基元（3DGS），四维高斯基元（4DGS）在空间坐标 $(x,y,z)$ 的基础上引入了时间维度 $t$。每个高斯基元可视为四维空间中的超椭球体，其均值向量表示为 $\mu = (\mu_x, \mu_y, \mu_z, \mu_t)$。其协方差矩阵 $\Sigma$ 由四维尺度矩阵 $S = \mathrm{diag}(s_x, s_y, s_z, s_t)$ 与由四元数 $q_l$ 和 $q_r$ 分别对应的左旋转矩阵 $L$ 和右旋转矩阵 $R$ 共同构成的四维旋转矩阵 $R$ 相乘得到，具体计算公式为：
\[
\Sigma = R S S^T R^T
\]
在给定时间变量 $t$ 的条件下，四维高斯分布的泼溅过程首先假设时间维度的边缘分布 $p(t)$ 服从一维高斯分布，即 $p(t) = \mathcal{N}(t; \mu_t, \Sigma_{4,4})$。基于此边缘分布，四维高斯分布 $p(x,y,z,t)$ 可分解为条件分布的形式：
\[
p(x,y,z,t) = p(t) \, p(x,y,z \mid t)
\]
其中，三维空间中的条件高斯分布 $p(x,y,z \mid t)$ 将用于后续的渲染过程。同时，四维高斯基元依据该条件分布投影至三维空间，其条件均值与协方差矩阵分别为：
\[
\mu_{xyz \mid t} = \mu_{1:3} + \Sigma_{1:3,4} \Sigma_{4,4}^{-1} (t - \mu_t)
\]
\[
\Sigma_{xyz \mid t} = \Sigma_{1:3,1:3} - \Sigma_{1:3,4} \Sigma_{4,4}^{-1} \Sigma_{4,1:3}
\]
随后，三维空间中的高斯基元被投影至二维屏幕平面。已知相机的内参矩阵 $K$ 及外参矩阵 $E$，三维高斯分布的中心点首先通过透视投影映射至像素坐标系，具体过程为：
\[
\mu^{2d} = \mathrm{Proj}(\mu_{xyz \mid t} \mid E, K)_{1:2}
\]
同时，利用透视投影的雅可比矩阵 $J$ 对协方差矩阵进行一阶近似，将三维协方差矩阵 $\Sigma_{xyz \mid t}$ 映射为二维协方差矩阵：
\[
\Sigma^{2d} = \bigl(J E \, \Sigma_{xyz \mid t} \, E^T J^T \bigr)_{1:2,1:2}
\]
因此，在屏幕平面上，每个四维高斯分布在时间点 $t$ 对应一个二维高斯分布 $p(u,v \mid t)$，从而在像素坐标 $(u,v)$ 处产生椭圆形的泼溅（splat）贡献。在获得时间分布 $p(t)$ 与二维空间分布 $p(u,v \mid t)$ 后，4DGS 将二者结合，得到时空域内的像素密度分布：
\[
p(u,v,t) = p(t) \, p(u,v \mid t)
\]
基于上述方法，实现了前向合成渲染。对于同一视角下在时间点 $t$ 可见的高斯基元，按照深度顺序进行排序后，像素 $(u,v,t)$ 的最终颜色由所有平面高斯基元的加权累积结果确定：
\[
I(u,v,t) = \sum_{i=1}^N p_i(t) \, p_i(u,v \mid t) \, \alpha_i \, c_i(d,t) \prod_{j < i} \bigl(1 - p_j(t) \, p_j(u,v \mid t) \, \alpha_j \bigr)
\]
其中，$\alpha_i$ 表示第 $i$ 个高斯分布的不透明度，$d$ 表示视线方向。为刻画外观随视角与时间的变化，4DGS 采用四维球柱谐波对函数 $c_i(d,t)$ 进行建模。该方法通过将时间维度上的傅里叶基与空间视角维度上的球谐基相乘，构建出一组完备的基函数，从而在统一的四维表示框架下，同时描述几何运动与外观的动态变化。

### 动态场景高斯渲染加速

近年来，针对 Gaussian Splatting 的高效推理需求，出现了两类主要的渲染加速技术：基于通用 GPU 的软件加速以及利用时间冗余的动态 3DGS /4DGS加速架构。表1展示了我们的方案与这些方案的对比。

| 方案          | 高斯表示       | GPU 渲染兼容性 | 访存级时间感知高斯剪枝 | 跨帧片上高斯复用      |
| ------------- | -------------- | -------------- | ---------------------- | --------------------- |
| 4DGS          | 原生 4DGS      | ✔（GPU）       | △（需加载到GPU）       | ✖（逐帧渲染）         |
| 4D-GS（Wu）   | 4D 体素 + 3DGS | ✔（GPU）       | △（需加载到GPU）       | ✖（逐帧渲染）         |
| 4DGS-1K       | 原生 4DGS      | ✔（GPU）       | △（仅仅离线剪枝）      | ✖（逐帧渲染）         |
| MEGA          | 基于 4DGS      | ✔（GPU）       | △（需加载到GPU）       | ✖（逐帧渲染）         |
| Neo           | 时间序列 3DGS  | ✖（ASIC）      | ✖（仅跨帧重排深度）    | ✖（仅排序表）         |
| LS-Gaussian   | 静态 3DGS      | ✖（ASIC）      | ✖（假定外观静态）      | ✖（仅图像域结果）     |

| GSPIM（本文） | 原生 4DGS      | ✔（GPU+PIM）   | ✔（PIM）               | ✔（高斯基元）         |

#### 软件级 GPU 高斯渲染加速

当前的软件级 4DGS GPU 加速方法主要聚焦于在通用 GPU 上提升动态场景的新视图合成性能。代表性研究包括：Yang 等人提出的 4DGS 方法，将空间与时间整体建模为原生 4D 高斯体积，并结合基于体渲染的可微 splatting 管线，在单卡 GPU 上实现了实时且高保真的动态场景渲染；Wu 等人的 4D-GS 采用“4D 神经体素 + 3D 高斯”的混合表示及分解式体素编码网络，将大量时空信息编码在紧凑的 4D 体素场中，通过条件化解码得到每一帧的 3D 高斯集合，显著降低了训练与存储成本，同时在单卡上实现了较高帧率；Zhang 等人的 MEGA 方法则在 4DGS 框架下从颜色与形变参数入手，引入共享 AC-color 预测网络与熵约束形变场，在保持渲染质量的前提下大幅压缩了 4D 高斯的存储与显存占用，使大规模动态场景的原生 4D 表示更加易于部署；近期的 4DGS-1K 工作则从原生 4DGS 表示中暴露出的时间冗余入手，通过基于时空变化评分的全局剪枝策略去除短寿命与低贡献高斯，并结合关键帧驱动的 temporal filter 只激活当前帧相关的高斯，在基本保持渲染质量的前提下显著压缩了高斯数量并提升了渲染效率。总体来看，这些工作分别从原生 4D 表示、混合 4D 表示与 3D 高斯、参数化改进以及模型压缩等多个维度，推动了动态场景在通用 GPU 上的实时高保真渲染。然而，它们在系统层面存在若干共性限制：一是时间信息主要服务于表示和网络建模，或在 GPU 内核中用于着色与条件化，未能充分消除软件层次筛选低时间贡献高斯基元带来的大量加载开销；二是渲染数据流通常还是逐帧渲染，对同一时间窗口内高斯的跨帧片上复用缺乏整体机制，仍以逐帧加载和处理为主。随着动态场景规模和时间长度的增长，这些缺失在带宽冗余和跨帧复用受限方面形成了明显的体系结构瓶颈。

#### 硬件级动态 3DGS/4DGS 加速器

现有针对 3DGS 时间序列渲染的加速器普遍通过显式挖掘连续帧间的时间冗余来提升吞吐量。Neo 面向排序瓶颈，观察到同一 tile 内相邻帧高斯的深度顺序高度相似，提出“reuse-and-update”增量排序策略：复用上一帧已排序的高斯列表，仅对视点变化导致的顺序失配以及新增/消失高斯局部执行动态部分排序，并配合专用排序单元显著降低重排算力和 DRAM 读写开销。然而 Neo 仍按逐帧静态 3DGS 建模，后续光栅化与着色流水线需要在每一帧基本重复执行，时间冗余被限制在排序阶段，未在时间维度上统一建模整体数据流。LS-Gaussian 在静态 3DGS 场景中利用帧间相似性，通过视点变换将上一帧颜色与深度重投影到当前视点，仅对重投影误差较大的 tile 触发稀疏重渲；同时基于视点变换预测 tile 负载，驱动集成 Load Distribution Unit 的流式加速器实现均衡调度与减少 stall。但其假设几何与外观随时间不变，时间轴仅体现为相机轨迹，不支持携带时间协方差与解析条件化算子的 4DGS 表示。3DGauCIM 则直接采用 4DGS 统一表示动态场景，通过 4D→3D 解析时间切片构建动态 3DGS 渲染管线，并在 DCIM 架构上引入多种跨帧优化：DR-FC 利用离线构建的粗粒度时空网格进行视锥裁剪以减少 DRAM 访问，ATG 跨帧记录高斯与 tile 的相交关系自适应生成 tile 分组提升片上缓冲复用，AII-Sort 以上一帧的深度分布初始化当前帧桶区间以获得更均衡的分桶。尽管如此，DD3D-Flow 仍按帧独立处理时间切片后的 3D 高斯集合，每帧都需重新从 DRAM 取回并处理当前帧全部活跃高斯；其跨帧复用仅停留在 tile 分组和深度区间等统计信息层面，未能充分利用跨帧高斯基元相似性来实现高效片上高斯基元复用。总体来看，这些加速器虽然充分挖掘了帧间时间冗余，在排序、光栅化等局部阶段实现了显著加速，但时间维度仍主要作为跨帧优化的指导信号，尚未将跨帧复用真正转化为减少 4DGS 渲染中重复加载高斯基元所带来的带宽开销。

### 近存计算

除了上述关于动态场景高斯渲染算法及其GPU实现优化的研究外，从体系结构视角出发，近存计算（Processing-in-Memory, PIM）为缓解数据密集型图形与视觉任务中的带宽瓶颈问题提供了新的解决方案。PIM的核心理念是在存储阵列附近集成可编程或专用的轻量级计算单元，使得部分传统上需在处理器端完成的计算能够在内存侧直接执行，从而显著减少处理器与高带宽存储之间频繁的数据传输，缩短访问路径并提升带宽利用效率。已有研究已在张量运算、图处理及静态场景高斯渲染等领域验证了PIM在算力充足但数据搬运受限的工作负载中的有效性，表明计算贴近数据的体系结构设计具有广泛的适用性。本文针对4D Gaussian Splatting推理过程中同样存在的带宽压力及跨帧数据冗余问题，提出在高带宽存储内部署面向高斯表示的PIM单元，并结合GPU渲染管线进行协同设计，构建整体加速架构。

## 4DGS加速架构设计的挑战

我们注意到使用GPU处理4DGS或动态3DGS的数据流具有两个关键特征：（i）预处理阶段通过时间贡献筛选高斯基元；（ii）在时间尺度上逐帧进行个顺序渲染。具体而言，数百万个训练好的4D高斯基元（通常以SoA方式存储）会根据其在当前时刻的时间贡献进行筛选，过滤掉贡献较小的高斯基元，随后通过视锥筛选和一系列矩阵运算，依次转换为3D和2D表示，并在时间尺度上进行渲染。该推理过程会重复多次，从而生成多帧视频。为了深入分析这种数据流，我们在搭载H100 80GB的环境下，采用先进的4DGS-1K的策略进行算法部署，对DyNerf数据集的多个小幅度动作场景，和ZJU EasyVolcap大幅度舞蹈场景进行了30000次迭代的训练，并选取最优检查点在固定测试相机下进行10秒300帧视频推理。在推理过程中，我们发现这种标准数据流存在两大制约硬件效率的局限性：

在标准的 4DGS 数据流中，预处理阶段通常是整个推理流程的起点。系统需要顺序加载并扫描所有四维高斯基元，从 HBM 依次取出其尺度参数 S 以及旋转矩阵 L 和 R，计算时间贡献函数 p(t)，并据此剔除时间贡献不超过 0.05 的高斯基元。然而，如图 3(c) 所示，在图 3(a) 的动态前景区域中，为刻画图 3(b) 所示跨帧变化较大的细节，不得不引入大量高斯基元；这些基元实际参与渲染的时间跨度往往较短（图 3(d)），在大部分帧中的时间贡献接近于零。与之相比，静态背景仅需少量高斯基元即可完成描述，这些基元在多帧中保持较高时间贡献、更为活跃，但总体数量远少于动态区域。在这种结构下，即便在采用类似 4DGS-1K 的离线静态时序剪枝后，用于完整描述整帧的高斯集合中，仍有相当比例在预处理阶段被时间贡献判定为当前帧不使用。如图 3(b) 的实验结果所示，超过 70% 的高斯基元在这一阶段被时间筛选拒绝，其对应的 HBM 访存从带宽角度来看几乎全部转化为访问冗余。考虑到处理器通常以 cache line 为最小批量加载数据，这种高筛选率会在预处理阶段引入两类显著的冗余访问：（i）为计算 p(t) 必须加载的高斯基元关键属性（约 52 字节）在被筛除后完全未产生有效贡献，属于纯粹的带宽浪费；（ii）其他属性（约 584 字节）由于空间相邻性而被一并带入缓存。由于活跃高斯基元在物理地址上往往与大量不活跃基元混排，相邻的不活跃基元也会被整体送入 L2 Cache，其附加属性数据流量同样全部消耗片外带宽却不参与后续渲染。我们基于 NVIDIA Nsight Compute 对预处理阶段的 HBM–L2 访存行为进行了采样分析，并将内存访问模式与算法语义结构进行关联。在不同动作幅度的动态场景中，如图 3(e) 所示，约有 75%–85% 的 HBM→L2 数据加载最终并未参与当前帧的有效计算。可以看到，在现有 4DGS 的“统一加载再软件筛选”策略下，大量本就不会用于当前时间窗口渲染的不活跃高斯基元仍然触发了包含其全部属性的完整访存，使得片外内存带宽的大部分被消耗在一次性的时间贡献验证上，这已经成为预处理阶段的主要性能瓶颈之一。

4DGS在表示层面展现出一种基本的统计特性，即时间上相邻帧之间共享高度重叠的活跃高斯基元集合。以图4所示的测量数据为例，在时间窗口大小为2至5帧的范围内，相邻帧活跃高斯基元集合的Jaccard相似度始终维持在较高水平。即便在动作幅度较大的场景（如dance3）中，5帧时间窗口内仍有超过75%的高斯基元能够跨帧复用并参与连续渲染。然而，现有4DGS渲染实现普遍采用以帧为基本单元的frame-major逐帧数据流处理方式：每一帧被视为独立的渲染任务，计算过程中需从HBM中遍历所有参与该帧渲染的活跃高斯基元。尽管当前许多工作（4DGS-1K）利用帧的相似性减少了相邻帧每帧的高斯基元加载量，但正如图4所示，我们在不同场景下的测量结果表明，在视频序列推理过程中，虽然随着时间窗口从2帧扩大至5帧，窗口内可复用的高斯基元数量有所减少，但平均每个可复用活跃高斯基元的重复加载次数却近乎线性增长（分别为34.5、52.2、70.2、88.5次），表明其未能在片上存储中实现有效复用。这主要归因于单帧渲染所需的活跃高斯工作集规模远超片上存储容量，导致帧间可复用的高斯基元在当前帧渲染完成后被缓存替换策略驱逐出片上存储；而在下一帧计算时，由于相邻帧所依赖的高斯基元高度相似，这些基元往往需重新加载。因此，4DGS基于帧间隔离的渲染数据流与处理器片上存储的组织方式不可避免地引发跨帧可复用高斯基元的频繁驱逐与重复加载，进而导致系统性片外带宽访问的能效损失。

### 我们的方法

为了解决现有GPU数据流的上述局限性并实现高效的4DGS推理，我们提出了GSPIM——一个基于GPU和PIM异构4DGS渲染加速架构。该架构具有两个关键的数据流创新：

#### 近存时间贡献筛选（Filter）

为避免基于时间贡献筛选导致高斯基元属性冗余加载的问题，我们提出了近存时间贡献筛选方案。该方案通过将时间贡献筛选阶段前移至 HBM，在高斯基元加载到GPU之前，利用近存计算在HBM中预先计算其时间贡献，实现了对原始 4DGS 数据流入口的结构性重构，从而有效消除因被时间贡献筛选掉的的高斯基元预处理引发的数据冗余迁移。

#### 跨帧高斯渲染流（Flow）

与传统的帧隔离处理方法不同，我们提出了一种新型数据流方案，实现了跨帧渲染中内存加载负载的有效降低。该方案在4DGS连续帧渲染的过程中，尽可能复用片上已加载的高斯基元，渲染多个相邻帧的部分内容，使得每个高斯基元仅需加载一次即可同时用于多帧渲染，从而显著减少了单帧平均高斯基元的加载量及整体数据搬运量。

GSPIM 数据流通过在内存端预先过滤无效的高斯基元，并在 GPU 端采用跨帧复用的批量渲染策略，有效解决了传统4DGS数据流中预处理阶段存在的冗余数据加载及跨帧渲染效率低下的问题，进而显著提升了带宽利用率和整体系统性能。需要注意的是，我们的方法主要面向时间尺度上的连续渲染，这些技术与相机位置改变的空间尺度3DGS实时渲染优化技术是正交的。

## GSPIM 数据流

GSPIM架构采用具有两个创新特点的数据流架构：近存时间贡献筛选预处理和跨帧高斯渲染。与传统的4DGS逐帧渲染数据流不同，本设计将在时间上连续的帧视为整体的时间窗口，利用片上可复用的高斯基元进行批量渲染，如图5所示。

![Design.drawio](./GSPIM.assets/Dataflow.drawio.svg)

### Stage 1: PIM Time Contribution Filter

GSPIM 的数据流首先在HBM端通过 PIM 单元计算每个高斯基元的时间贡献 $p(t)$，以便在数据进入 GPU 之前，提前筛选出在当前时间窗口内基本无效的高斯基元。时间贡献的估计依赖于高斯基元的旋转矩阵 $L$、$R$、当前时间戳 $t$、时间尺度均值 $\mu_t^{(4)}$和尺度向量$\bold{s}$。具体而言，PIM 单元在 HBM存储器中并行读取多个高斯基元的相关参数，计算出4D协方差矩阵中时间的自相关系数$\Sigma^{(4)}_{t,t}$，进而获得对应的时间贡献值 $p(t)$。这一步的详细计算策略参见下一章。

鉴于 GSPIM 采用了在时间窗口 $[t_{\min}, t_{\max}]$ 内的多帧联合渲染策略，系统需对每个高斯基元在该时间窗口内的最大时间贡献值 $p_{\max}(t)$ 进行判定，并据此决定是否保留该高斯基元。考虑到时间贡献函数 $p(t)$ 服从均值为 $\mu^{(4)}_t$ 的一维高斯分布，其最大值必然出现在最接近均值 $\mu^{(4)}_t$ 的时间点。当 $\mu^{(4)}_t\in[t_{\min}, t_{\max}]$ 时，高斯函数在窗口内部达到峰值，此时PIM将计算 $p(\mu_t)$作为$p_{\max}(t)$；反之，若 $\mu^{(4)}_t$ 不在该区间内，则最大值位于时间窗口的边界处，PIM将计算 $p(t_{\min})$ 或 $p(t_{\max})$ 作为 $p_{\max}(t)$。

若某高斯基元的$p_{\max}(t)$ 低于预设阈值（例如 0.05），则 PIM 会在内存中构建一张Active Map，标记这些无效的高斯基元，并由PIM-Side Active Loader将这些活跃的数据搬运到HBM中预留的紧凑区域， GPU 加载时只会访问该活跃紧凑基元，避免对低活跃度的基元的访问。该基于时间统计特性的近存筛选机制有效避免了大量对当前时间窗口无贡献的高斯基元被完整加载至 GPU，显著降低了冗余带宽开销，并为后续跨帧批次化渲染构建了更为紧凑的有效工作集。

### Stage 2: Multi-Frame 4D Projection

在PIM完成对无效高斯基元的筛选后，GPU能够对当前视窗内实际参与渲染的高斯基元执行从四维到三维的投影操作。具体而言，GPU首先利用四维旋转矩阵、尺度向量及当前时间戳计算完整的四维协方差矩阵$\Sigma^{(4)}$。随后，基于时间维度对该协方差矩阵及对应的四维均值$\mu^{(4)}$进行条件化处理，进而转换为三维协方差矩阵$\Sigma^{(3)}$和三维均值$\mu^{(3)}$，以此确定每个四维高斯基元在三维空间中的形状特征及中心位置。

与传统逐帧独立完成的4D到3D投影方法不同，Stage 2以时间窗口为基本处理单元，能够并行生成时间区间$[t_{\min}, t_{\max}]$内所有目标帧的三维表示，从而使系统在单次数据加载周期内完成多帧的条件化投影操作。渲染窗口的大小$W$并非固定值，而是根据输入序列的时间动态特性自适应调整：当场景随时间变化较为平缓时，大多数高斯基元的深度变化较小，系统倾向于采用较大的时间窗口，以提升帧间深度相近的高斯基元在单批次中的复用效率；反之，若场景中存在快速运动或几何结构突变等情况，时间窗口则会自动缩小，以降低快速变化的高斯基元跨批加载对复用效率的负面影响。

### Stage 3: Depth Analysis &Stability-Based Classification

在 Stage 2 获得时间窗口 $[t_{\min}, t_{\max}]$ 内所有帧的三维中心点后，Stage 3 将这些中心点投影到相机坐标系，计算每个高斯基元在该时间窗口内的深度序列 $\mathcal{D} = \{d_1, d_2, \dots, d_W\}$。传统 4DGS 通常对每帧所有高斯基元独立执行全局深度排序以确保遮挡关系准确，但跨帧重复排序导致计算冗余。我们发现，大多数自然场景中，相邻帧间高斯基元深度变化平稳，只有在物体快速运动或遮挡突变时，少数基元深度才显著跳变。

基于上述观察，我们提出用深度稳定性分数衡量高斯基元在时间窗口内的深度稳定性。在profilling的过程中，我们发现大量的高斯基元的深度稳定性都是很高的，因此我们采用能够在归一化区间内用较大值变化反映变量变化的指数衰减函数来计算深度稳定性分数。首先，我们对每个高斯基元的深度序列$\mathcal{D}$求取其变化因子$\Delta D$：
$$
\Delta D = \sum_{i=1}^{W-1}|d_{i+1}-d_i|
$$
随后，利用指数衰减函数计算其深度稳定性分数（$SS$）：
$$
SS = \exp\{-\Delta D\}
$$
当 $SS$接近1时，深度分布高度集中，表明跨帧深度稳定；接近 0时，深度分布分散，存在显著跨帧跳变。基于此，设阈值 $SS > \tau_s$（如 $\tau_s=0.95$），将高斯基元分为深度稳定和不稳定集合。深度稳定基元在Stage 4中统一处理，不稳定基元则需单帧精度额外处理，以确保遮挡关系准确。

深度稳定性分数不仅用于基元级分类，还能反映窗口内整体动态复杂度。当所有高斯基元的平均深度稳定性分数低于阈值 $\tau_w$（如0.9）时，表明场景中存在大规模快速运动或遮挡变化。此时，继续使用较大时间窗口会显著增加不稳定基元数量，降低跨帧复用效率。为此，系统将在下一轮推理中自动缩小时间窗口宽度 $W$，以适应快速变化的动态场景；反之，平均稳定性分数较低时则适当扩大窗口宽度，提升跨帧复用率。

### Stage 4: Sorting & Batch Scheduling

在基于深度稳定性分数完成高斯基元分类后，系统需为时间窗口内的每一帧构建严格一致的渲染顺序。对于深度稳定性较低、跨帧深度变化幅度极小的基元，系统根据其在窗口中的最小深度 $d_{\min}$ 执行一次全局排序；这一排序能够在多个时间点间保持遮挡关系的稳定性，因此可以被直接共享至窗口中的所有帧。相反，对于深度稳定性分数较高、跨帧深度跳变显著的基元，其深度序列在时间维度上表现出强烈的非平稳性，若仍沿用全局顺序将不可避免地引入遮挡错误。因此，系统需在每帧中独立执行局部深度排序，以确保这些不稳定基元在各自帧中的前后关系严格正确。随后，系统将稳定集合的全局排序结果与不稳定集合的逐帧排序结果进行归并，构建出时间窗口内所有帧的最终渲染顺序列表（ROL）。基于该跨帧一致的渲染顺序，GSPIM 进一步需将全部高斯基元划分为若干批次，并按照批次顺序在时间窗口内加载、驻留与替换，以保证每一批次在其驻留周期内完成相应的多帧渲染任务。

批次划分的首要目标是维持跨帧渲染顺序的正确性；在满足该约束后，批次规模即可退化为单纯的硬件容量约束问题。具体而言，L2 Cache 中需要同时驻留当前批次的全部高斯基元，并保证其二维投影参数、椭圆矩阵及颜色贡献在连续多帧渲染中的重复访问不发生缓存抖动。因此，决定批次规模的并非 L2 Cache 的全部容量，而是扣除光栅化中间结果、调度元数据及其他系统固定占用后，可供分批高斯参数驻留的有效缓存空间。我们将这部分可被稳定复用的缓存容量定义为 $L_2^{\mathrm{eff}}$，并在后续的双缓冲调度机制中将其划分为当前批次驻留区与下一批次预取区，使渲染计算与数据加载能够持续重叠。

在已知 $L_2^{\mathrm{eff}}$ 的前提下，当前批次可使用的驻留容量 $C_{\text{cur}}$ 决定了单批能容纳的高斯基元数量。设单个高斯基元在 L2 中的平均驻留占用为 $s_g$（包括缓存行对齐开销与因临近访问产生的冗余字节），则批次大小$B$可建模为：
$$
B = \left\lfloor \frac{C_{\text{cur}}}{s_g} \right\rfloor_{\mathrm{warp}}
$$
其中下标 $\mathrm{warp}$ 表示对批大小进行 Warp 对齐，以确保在 GPU 的执行粒度下不会产生资源碎片和线程发散。该批次规模不仅决定了系统在当前时间窗口中能够形成的批数，也决定了高斯基元在跨帧渲染中的驻留时长，从而直接影响跨帧复用效率与整体渲染吞吐率。在既定的批大小下，系统将结合高斯基元在时间窗口中的有效渲染区间进一步组织批次顺序，使其在满足深度依赖关系的前提下最大化跨帧驻留收益。

### Stage 5: Batch-Major Multi-Frame Rendering

在构建了适应显存驻留的批次结构后，Stage 5 在 GPU 上以batch-major的方式执行多帧渲染。渲染过程依然遵循3D Gaussian Splatting的通用流程：每个高斯基元在 3D 空间中的均值和协方差被映射为二维椭圆参数，其颜色、透明度及对像素的贡献通过椭圆区域内的重叠程度进行计算，并按照深度顺序完成前向合成。为实现高吞吐量，系统采用基于 GCC 的 tile-based 并行光栅化策略，使批次中所有基元在渲染多帧时共享同一套 tile 结构，并利用透射率提前终止机制加速合成过程。

得益于 Stage 4 中采用的 batch-major 调度策略，每个批次在显存驻留期间能够连续为多个帧提供渲染输入，从而实现了高斯基元在跨帧渲染中的复用，避免了频繁的重新加载操作。同时，基于深度稳定性的差异化处理方法在绝大多数场景下有效维持了渲染顺序的一致性及几何结构的准确性，仅对深度稳定基元引入了轻量的顺序近似，而这类近似在视觉上通常可以忽略。该阶段最终将多帧批次的渲染过程整合为连续的视频输出，为 GSPIM 在加速 4DGS 推理中的整体性能提升提供了关键支持。

## GSPIM Architecture

![Arch.drawio](./GSPIM.assets/Arch.drawio.svg)

### Overview

GSPIM 采用由 CPU、GPU 及 PIM-HBM 协同构成的异构系统架构（见图 6），以支持时间贡献筛选与跨帧批量渲染在硬件体系结构层面的高效执行。GPU 作为主要计算核心，集成多组SM阵列、片上共享内存、L1/L2 缓存以及通向HBM的高速接口，负责完成模型在前述数据流中从四维到三维的条件化投影、深度与可见性分析、批次排序及多帧光栅化等高度并行的计算任务。与 GPU 相连的 HBM 被扩展为集成轻量级PIM的 PIM-HBM 结构：这些 PIM 单元部署于各memory bank侧，支持定点乘加等基础算术运算，并可通过InterConnect在不同计算阶段复用资源。该设计使系统能够直接在 HBM 内完成高斯基元时间贡献上界的估计Active Map的生成，从体系结构层面有效压缩传输至 GPU 的数据规模。

基于上述基础，GSPIM 的执行路径体现为面向架构边界的数据筛选与计算复用的协同机制：CPU 负责时间窗口的管理、批次调度决策以及向 GPU 和 PIM 发出控制指令；PIM-HBM 根据 CPU 指定的时间窗口配置，在内存端完成高斯基元的时间贡献评估与无效基元的剔除，仅将对当前时间窗口帧渲染有效的高斯基元数据块通过 HBM–GPU 高带宽互连传输至 GPU；GPU 则在片上缓存与 HBM 显存之间构建批次化的驻留与替换机制，使同一批高斯基元在其生命周期内能够服务多个相邻帧的渲染请求，并在 L2 Cache 中形成可跨帧复用的高局部性工作集。通过在 CPU、GPU 与 PIM-HBM 之间重新划分计算职责与数据传输路径，GSPIM 在不改变上层 4DGS 接口的前提下，实现了时间贡献筛选与跨帧渲染在体系结构上的自然映射，从而系统性地提升了带宽利用率、缓存效率及推理能效。

### PIM-HBM Design and Data Layout

为在数据流早期剔除对当前时间窗口贡献极低的高斯基元，GSPIM 在 HBM 的每个 bank 侧集成了一个轻量级 PIM 单元，使得部分空间–时间相关计算能够在内存端直接完成。该单元以 16 位整数部分与 10 位小数部分组成的 26 位定点格式作为核心数制，支持从 FP16 半精度浮点至定点的数制转换，并在定点域执行整个时间边缘概率的计算链条。PIM 单元由可重构的算术功能单元（FU）、长度为 16 的定点寄存器文件以及有限状态机驱动的控制模块构成。其中，FU 内包含 4 个定点乘法器与 3 个定点加法器，可通过片内 Interconnect 以阶段粒度重新配置，实现对同一算术资源的跨阶段复用；Regfile 则在各阶段之间保存关键中间量；控制模块根据执行阶段调度 Interconnect 的连线方式，从而保证整个时间贡献验证过程在 HBM 内顺序展开。通过这种阶段级算力复用方式，PIM 单元在有限面积与能耗预算下，仍能完成构建空时协方差所需的核心线性代数计算。

为降低近存时间贡献计算的访存复杂度，并提升活跃高斯基元在 PIM-HBM 侧的紧凑化效率，我们对高斯基元的原始数据布局进行了结构化重组，如图 X 所示。在单个 Die 内，我们预留约 25% 的 Bank 空间作为紧凑化缓冲区，用于存放经 Active Loader 重排后的活跃高斯基元；其余 Bank 则维持原始数据存储区域。传统 4DGS 数据结构为了适配 GPU 的连续加载需求采用 SoA布局。受此影响，本应属于同一高斯基元的 52 字节时间相关字段被分散映射至不同 Bank 行，导致在 PIM-HBM 中执行 $p(t)$ 计算时必须跨行、跨 Bank 访问，不仅增加延迟，也降低了行缓冲命中率。为此，我们在 HBM 中重新组织了这些时间相关字段，使其在单个 Bank 行内连续排布，使 PIM 单元能够通过一次行激活将所需的全部时间相关数据加载至行缓冲，显著简化了活跃性判定过程的访存路径。此外，为进一步降低紧凑化阶段的数据搬运成本，我们要求同一批次高斯基元的全部属性字段必须映射至同一个 Bank Group 内。此约束保证了 PIM 侧在根据 Active Map 选取活跃高斯并执行属性搬运时，无需跨 Rank 或跨 Die 访问，大幅减少访存冲突和内部互连开销，同时使得紧凑数据能够在 Die 内高效、连续地构建。

### PIM Dataflow

![PIMFlow.drawio](./GSPIM.assets/PIMFlow.drawio.svg)

图7展示了GSPIM在近存侧完成时间贡献验证的五阶段计算流程。该流程以构建时间维协方差项$\Sigma_{3,3}$为核心，基于四维高斯协方差的尺度-旋转分解结构，从四元数诱导的旋转矩阵中提取与时间维相关的最小计算单元。为最大化近存端的算力与带宽效率，每个阶段均对原始矩阵运算进行了结构化裁剪，使计算严格聚焦于$\Sigma_{3,3}$所需的行列展开。同时，通过阶段级复用充分利用PIM功能单元中的乘法器和加法器，以最低计算代价实现时间边缘概率的近存判定。

#### Stage 1：Temporal Offset and Scale

数据流首先在 PIM 内完成高斯参数的统一数值化和时间偏移的前置计算。PIM 从 HBM 以字段粒度读取高斯基元的必要参数，并将其转换为 26 位定点格式，以确保后续矩阵操作在近存乘加路径上的数值稳定性。随后，根据尺度参数构建四维尺度矩阵$S=\mathrm{diag}(s_x,s_y,s_z,s_t)$并依据 GSPIM 的时间戳选择策略（在 $\mu_t, t_{\min}, t_{\max}$ 中选取最接近 $\mu_t$ 的点）计算时间偏移量 $d_t$。

#### Stage 2：Construction of the 4D Rotation Matrix

数据流首先在PIM内部完成高斯参数的统一数值化处理及时间偏移的预计算。PIM以字段为粒度，从HBM中读取高斯基元所需参数，并将其转换为26位定点格式，以确保后续矩阵运算在近存储乘加路径上的数值稳定性。随后，基于尺度参数构建四维尺度矩阵$S = \mathrm{diag}(s_x, s_y, s_z, s_t)$，并依据GSPIM的时间戳选择策略（即在$\mu_t, t_{\min}, t_{\max}$中选取最接近$\mu_t$的时间点）计算时间偏移量$d_t$。

#### Stage 3：Scaled Rotation Factor

在获得旋转矩阵与尺度矩阵在时间维度上的相关行列后，本阶段构建尺度-旋转因子 $M = S R$ 的时间相关切片。由于 $S$ 为对角矩阵，PIM可通过逐元素乘法直接计算出 $M$的第四行（由 $S_{3,3}$ 与 $R_{3,0:3}$相乘得到）及第四列（由各对角元素 $S_{ii}$与 $R_{0:3,3}$相乘得到）。该过程仅需两轮四元素并行乘法，便能提取并构造出 $\Sigma_{3,3}$所需的全部矩阵片段，从而有效避免了在近存储器中执行完整矩阵乘法所带来的高昂计算开销。

#### Stage 4： Temporal Covariance Evaluation

在获得矩阵$M$的关键行列之后，第四阶段基于协方差分解式$ \Sigma = M^\top M$计算时间维协方差项$\Sigma_{3,3}$。该协方差项等价于矩阵$M$在时间维度上对应的行向量$M_{3,0:3}$与列向量$M_{0:3,3}$的内积。PIM通过四个乘法器并行计算对应元素的乘积，并利用加法规约树结构对乘积结果进行累加，从而得到协方差项$\Sigma_{3,3}$。

#### Stage 5：Time Contribution Filter

在最后阶段，我们利用已计算得到的 $\Sigma_{3,3}$ 实现了时间边缘概率的低成本阈值比较。原判定条件 $\exp\left(-\frac{0.5 d_t^2}{\Sigma_{3,3}}\right) < 0.05$ 在 PIM 中等价转化为 $0.5\times d_t^2 > \Sigma_{3,3}\times \ln(20)$，从而将指数运算和除法运算替换为两次乘法及一次比较操作。该计算结构可完全由 PIM 内部的两个乘法器和一个加法器实现，计算结果随后写回至 HBM 中的 Active Map。通过此计算方法，GPU 无需再将全部高斯基元从 HBM 加载至片上存储进行验证，即可明确识别出对当前帧渲染有效的高斯基元，从而显著降低了预处理阶段高斯基元的加载开销。

### PIM-Side Active Loader

在完成行缓冲时间贡献筛选并生成Active Map后，高斯基元仍以固定的全局顺序存储于HBM中。若GPU直接依据原始表格按L2缓存行大小顺序加载活跃高斯基元，必然会将大量邻近但不活跃的基元一并载入L2缓存，导致显著的带宽浪费及缓存污染。为避免此类问题，GSPIM在GPU访问启动前，于HBM内部对当前时间窗口内的活跃四维高斯基元进行近邻稠密化处理。基于此机制，我们在PIM-HBM架构中设计并实现了独立于GPU的Active Loader模块，使所有活跃高斯基元在传输至GPU之前，均被重新紧凑地安置于HBM预留的集中区域，从根本上消除GPU端的无效内存访问开销。

Active Loader部署于每个Memory Die内部，位于该Die各Bank之间的逻辑层，负责维护该Die内以SoA方式存储的各属性列表（除时间相关字段外）的起始地址表。当某Bank行缓冲区中所有高斯基元的时间贡献计算完成后，生成的Active Map将被传送至同一Die的Active Loader。该Loader依据属性起始地址表及当前高斯基元的物理顺序编号，精确计算每个活跃高斯基元在本Die内SoA分片中的内存地址，并通过Die内部数据总线向对应Bank发起读请求。获取完整属性后，Active Loader利用内部稀疏计数器为每个活跃基元分配预留Bank区域中的写入位置，将各字段写入紧凑的内存布局中；同时，原本以类似AoS形式排列于同一Bank行的52字节时间相关字段，在此过程中被转换回SoA形式，以便后续GPU端的矢量化访问。该稠密化策略使得不同Die内部预留的紧凑区域在地址空间中自然形成物理地址连续的Active Buffer。GPU在随后的可见性排序、深度组织及光栅化阶段，仅需顺序遍历该紧密排布的Active Buffer，而无需扫描原始高斯基元表，既避免了加载无关基元，又显著减少了GPU与HBM之间的无效数据传输，降低了片上L2缓存的无效访问率。

## GSPIM Scheduling

### 任务卸载与交互

在 GSPIM 调度框架中，编译器首先基于统一的4DGS算子描述，分别生成 GPU 端的 CUDA 内核代码与 PIM 端的可执行指令流。运行时，CPU 负责将任务卸载至 GPU。整个执行流程始于Stage 1：GPU 在当前时间窗口 $[t_{\min}, t_{\max}]$ 内，依据前述时间访问模型计算 $ \arg\max_{t} p(t)$，并将由该结果生成的 PIM 操作指令直接下发至高带宽存储器（HBM），以启动该时间窗口内的时间贡献处理。当 PIM 在 HBM 中完成时间贡献评估并返回该窗口的Active Map后，GPU 立即基于该 Active Map 对仍具有较高时间贡献的高斯基元进行选择性加载，随后依次执行Stage 2–Stage 4的投影、遮挡关系推断及特征重排等预处理步骤，同时在 GPU 内构建渲染前的打包结构。 

在 GPU 利用当前 Active Map 进行处理的同时，系统同步将下一时间窗口的时间贡献筛选任务卸载至 PIM，并将新一轮的 PIM 指令发送至 HBM，从而实现跨时间窗口的流水线并行处理——PIM 负责计算下一窗口的时间贡献，而 GPU 则处理当前窗口的投影与打包。最终，在渲染阶段，GPU 严格按照打包结构所确定的批次顺序，以分批加载的方式将高斯基元调入显存，完成多帧复用式渲染。每完成一个批次的多帧渲染后即将其换出，确保跨帧访问的一致性及显存占用的可控性，从而实现高效的渲染流程。

### GPU 分批高斯处理的计算-访存重叠

为实现分批高斯基元在单一时间窗口内的持续加载与高效渲染，GSPIM 在架构层面引入双缓冲机制，使 GPU 计算与 HBM 访存之间形成稳定且长期可维持的流水化执行路径。在上一阶段确定的有效缓存空间 $L_2^{\mathrm{eff}}$ 中，本阶段将其进一步划分为两个互补区域：用于驻留当前批次的 $C_{\text{cur}}$ 与用于预取下一批次的 $C_{\text{next}}$。其中，$C_{\text{cur}}$ 专门存放当前批次的 $B$ 个高斯基元，以保证其在多帧渲染中的高局部性访问；而 $C_{\text{next}}$ 则提前加载下一批次的数据，从而在空间层面实现深度重叠与无缝批次切换。

Profiling 结果显示，单个高斯基元在渲染阶段约占用 52 字节，而其对应的光栅化中间结果约需 28 字节。若当前批驻留区与其他数据区在 L2 缓存中长期维持 52:28 的稳定比例，则可显著降低缓存冲突率，减少缓存行频繁回收带来的性能抖动，并在批内光栅化阶段显著提升缓存命中率。为了确保下一批次在当前批渲染结束前能够完全预取到位，我们依据渲染阶段的有效计算吞吐率与 HBM 可用带宽之间的性能匹配关系，推导出当前批与预取区之间的容量约束：

$$
\frac{C_{\text{cur}}}{C_{\text{next}}} \approx \frac{TP_{\text{render}}}{B_{\text{HBM}}}
$$

其中 $TP_{\text{render}}$ 表示目标 GPU 上单批次渲染阶段的有效吞吐率，$B_{\text{HBM}}$ 表示可用的 HBM 带宽。实际应用中，我们根据目标 GPU 的 Profiling 结果，选择满足上述约束的容量配置，并将其固化为平台级调度参数，以确保在不同模型和场景下维持稳定的执行表现。基于这些信息，结合 GPU 计算与访存重叠的特点，能够针对具体硬件平台，确定前文所述的高斯基元分批大小 $B$。

当当前批次在 L2 缓存中被完全消耗后，下一批次的预取区通过连续地址空间的自然增长，直接转化为新的当前批次驻留区；与此同时，新一轮预取任务从后续地址区间开始填充，从而沿着物理地址顺序自动推进双缓冲结构。整个过程无需复杂的元数据迁移，也无需在 GPU 端执行额外的数据调度逻辑。在整个时间窗口内，当前批次的渲染计算与下一批次的数据预取始终保持严格重叠，构成了一个由数据布局自然驱动、可长时间稳定运行的双缓冲流水线。由此，分批高斯处理的整体吞吐率显著提升，同时避免了传统逐批加载模式下频繁出现的带宽峰值和缓存抖动问题。

### 基于全局高斯基元深度稳定性分数的窗口大小调整

为了适应 GSPIM 中随场景动态变化而调整的时间窗口策略，本文在渲染流水线中引入了一种基于跨帧统计特性的窗口自适应机制。当 Stage3 的投影与排序阶段完成后，CPU 通过一次异步数据拷贝获取 GPU 端输出的全部中间结果，并在 GPU 继续执行当前窗口渲染任务剩余时间的并行阶段，完成平均稳定性分数的计算。该分数度量反映了当前时间窗口内跨帧内容变化的稳定性：较低的分数表明相邻帧之间结构相对平稳，适宜扩展时间窗口以提升跨帧复用率及批处理利用率；反之，当深度稳定性分数超过预设阈值时，表明场景深度变化剧烈，继续扩大窗口将导致排序开销增加及重复加载频率上升，因此需收缩窗口以缓解因时间维度扩展带来的资源压力。CPU 在完成阈值判断后，将新的窗口配置以异步方式通过 PCIe 通道回传至 GPU，使其在下一窗口调度开始前获得更新参数，并据此同步调整 PIM 的时间贡献计算范围及自身的分批加载、打包与渲染策略。该机制确保 GSPIM 能够在不同运动速率及场景结构变化条件下，实现窗口大小的连续、低开销且稳定的动态适配，从而整体提升系统的时空一致性与执行效率。

## Evaluation

### Methodology

#### Baseline 

使用软件层最为SOTA的4DGS-1K论文的策略，把4DGS推理部署到单张NVIDIA H100 80GB@1755MHz显卡，作为baseline，基于此评估本文提出方法的有效性。

#### Benchmark

为评估所提方法在不同动作幅度场景下的加速性能，选取了动作幅度较小的Dynerf数据集和动作幅度较大的ZJU EasyVolcap数据集，分别对各自的300帧进行推理测试。Dynerf数据集采用1920×1080分辨率，并基于原作者开源仓库中的配置文件进行训练；ZJU EasyVolcap数据集则依据本研究调优后的配置进行训练。为确保训练效果，所有数据集均采用VGGT结合COLMAP方法构建初始点云。此外，每个模型均通过4DGS-1K的全局剪枝策略生成小规模版本，以便评估GSPIM在4DGS-1K算法框架下相较于纯GPU方案的加速效果及精度表现。

#### Experiment Platform

所有实验的GPU端均由单张NVIDIA H100 80GB（频率为1755MHz）GPU实现。GPU端采用CUDA实现了原始4DGS与4DGS-1K渲染管线，并在固定频率下运行，以消除动态电压频率调节（DVFS）带来的影响。PIM端基于Samsung PIMSimulator构建了时钟精确的PIM-HBM模型，依据H100所采用的HBM通道数及时序参数，配置其带宽与时钟频率。在运行GSPIM时，主机端通过软件代理将标记为PIM执行的内存访问及计算请求重定向至PIMSimulator，由模拟器提供周期级的执行时间与带宽利用率数据；随后，将该PIM执行时间与实际GPU渲染时间合成，以获得端到端的延迟指标。
